## 2. 架構問題

### 2.1 ❌ 無模組系統（嚴重）

所有 JavaScript 檔案都以 global scope 方式透過 `<script>` 標籤載入。這導致：

- **命名空間污染**：所有變數都在 `window` 上（`STEMS`、`BRANCHES`、`ELEMENT_CYCLE`、`HIDDEN_STEMS` 等）
- **載入順序依賴**：`index.html` 第 214-219 行必須按照固定順序載入
- **無封裝**：任何腳本都可以修改其他腳本的內部資料

```html
<!-- 目前的載入方式 — 順序不可變 -->
<script src="js/bazi.js"></script>
<script src="js/lunming.js"></script>
<script src="js/lunming2.js"></script>
<script src="js/geju.js?v=2"></script>
<script src="js/geju-ref.js?v=2"></script>
<script src="js/app.js?v=2"></script>
```

**建議**：使用 ES Module（`type="module"`），或以 Webpack / Vite 打包。

### 2.2 ❌ 全域狀態耦合（嚴重）

使用 `window._lastResult` 和 `window._lastGejuResult` 作為跨模組通訊的唯一管道：

```javascript
// app.js 第 287 行
window._lastResult = result;
// app.js 第 303 行
window._lastGejuResult = determineGeju(result);
```

這種方式讓資料流難以追蹤，且可能被任意程式碼修改。

**建議**：使用集中式狀態管理（簡單的 singleton Store 或 `WeakMap`）。

### 2.3 ❌ 無依賴注入

`geju.js` 的 `getTenGod`、`getStemIndex` 等函數依賴 `bazi.js` 的全域變數，但透過「若不存在則 fallback」的方式處理：

```javascript
// geju.js 第 722 行
function getStemIndex(stem) {
    if (typeof STEMS !== 'undefined') return STEMS.indexOf(stem);
    return ['甲','乙','丙','丁','戊','己','庚','辛','壬','癸'].indexOf(stem);
}
```

這是一種脆弱設計，隱藏了真正的依賴關係。

**建議**：將共用函數提取到獨立的 `utils.js` 或直接匯入模組。

### 2.4 ❌ 無架構分層

目前所有程式碼混雜在一起：
- 資料層（天干地支表）與計算層（排盤引擎）未分離
- UI 渲染層（`renderResults`）與業務邏輯層（`calculateBazi`）混合
- 配置（權重、分數閾值）散落在各處

## 3. 程式碼品質問題

### 3.1 ❌ 變數宣告不一致（中等）

- `bazi.js`：使用 `const`、`let`
- `lunming.js`：使用 `var`
- `lunming2.js`：使用 `var`
- `geju.js`：使用 `const`、`let`
- `geju-ref.js`：使用 `const`
- `app.js`：使用 `const`、`let`

**建議**：統一使用 `const` 為預設，只在需要重新賦值時使用 `let`，全域不使用 `var`。

### 3.2 ❌ 魔術數字散落各處（中等）

權重、分數、閾值直接寫死在程式碼中：

```javascript
// bazi.js 第 567 行 — 權重硬編碼
weight: 2   // 天干權重
weight: 1.5 // 地支權重
weight: i === 0 ? 0.8 : (i === 1 ? 0.5 : 0.3)  // 藏干權重

// lunming2.js 第 58-64 行 — 月令分數硬編碼
'木': { spring: 40, summer: 16, season: 8, autumn: 0, winter: 32 }

// bazi.js 第 646-648 行 — 身強/身弱閾值
const isStrong = selfStrength > otherAvg * 1.3;
const isWeak = selfStrength < otherAvg * 0.7;
```

**建議**：將所有配置常數集中在一個 `config.js` 檔案中，加上說明註解。

### 3.3 ❌ 函數過長（中等）

部分函數超過 100 行，難以測試和理解：

| 函數 | 位置 | 行數 |
|------|------|------|
| `calculateBazi` | bazi.js:863 | ~96 行 |
| `calculateGreatLuck` | bazi.js:757 | ~92 行 |
| `calculateDayMasterStrength` | lunming2.js:323 | ~168 行 |
| `determineGeju` | geju.js:258 | ~285 行 |
| `renderLunming2` | lunming2.js:780 | ~72 行 |
| `handleSubmit` | app.js:252 | ~72 行 |

**建議**：將超過 50 行的函數拆分為多個小函數（單一職責原則）。

### 3.4 ⚠️ 納音計算使用 Brute Force（輕微）

```javascript
// bazi.js 第 530-535 行
for (let i = 0; i < 60; i++) {
    if (i % 10 === stemIndex && i % 12 === branchIndex) {
        idx = i;
        break;
    }
}
```

有直接的數學公式可計算六十甲子索引，不需要迴圈。

**建議**：使用公式 `(stemIndex % 10) * 6 + (branchIndex % 12) - (stemIndex % 10) * ...` 的簡化版，或直接查表。

### 3.5 ⚠️ 節氣計算使用近似公式

`getLichunDate` 和 `getApproximateSolarTermDate` 使用簡化的天文近似公式，可能偏移 1-2 天。對於命理應用，這可能導致年柱/月柱判斷錯誤。

**建議**：
- 對於要求精確的應用，建議引入天文計算庫或內建節氣表
- 或至少加上「±1 天誤差」的免責提示

## 4. 效能問題

### 4.1 ❌ DOM 操作無快取（嚴重）

在 `renderLunming2`、`renderGejuTab` 等函數中，每次呼叫都重新查詢 DOM：

```javascript
// app.js 第 689 行 — 每次切換分頁都查 DOM
const root = document.getElementById('gejuRoot');

// lunming2.js 第 781 行
var root = document.getElementById('lunming2Root');
```

**建議**：在初始化時快取 DOM 參考（app.js 已部分做到，但其他模組未做）。

### 4.2 ❌ 每次渲染都重建全部 HTML（中等）

所有渲染函數都使用 `innerHTML =` 從頭建構 HTML 字串，沒有使用 DocumentFragment 或差異化更新。

**建議**：對頻繁更新的區塊，考慮使用模板字面量搭配 `insertAdjacentHTML` 或輕量框架。

### 4.3 ❌ 節氣日期重複計算（中等）

`calculateGreatLuck` 中對每一年都呼叫 `getSolarTermDates`，該函數又建立 12 個 Date 物件：

```javascript
// bazi.js 第 772 行
for (let y = birthYear - 1; y <= birthYear + 1; y++) {
    const dates = getSolarTermDates(y);
}
```

若使用者多次排盤，這些計算會重複執行。

**建議**：加入簡單的 memoization（快取），以 year 為 key 快取節氣日期。

### 4.4 ❌ 大運迴圈無上限（輕微）

`calculateGreatLuck` 中 `luckCycles` 固定產生 8 個 10 年週期，但若使用者年齡超過 80 歲則不足。

**建議**：根據使用者年齡動態計算所需週期數，或提供「載入更多」按鈕。

## 5. 資料重複與一致性

### 5.1 ❌ 核心資料重複定義（嚴重）

以下資料在多個檔案中重複定義：

| 資料 | 出現位置 | 
|------|---------|
| `HIDDEN_STEMS` 藏干表 | `bazi.js:38`、`lunming2.js:38`（不同格式）、`geju.js:34`（不同格式） |
| `STEM_ELEMENT` 天干五行 | `bazi.js:26`、`lunming2.js:22`（物件格式） |
| `BRANCH_ELEMENT` 地支五行 | `bazi.js:32`、`lunming2.js:31`（物件格式） |
| `ELEMENT_CYCLE` 五行循環 | `bazi.js:75`、`geju.js:733`（透過函數回傳） |

這導致：
- 修改其中一處容易忘記修改另一處 → 資料不一致
- 增加維護負擔
- 浪費記憶體

**建議**：建立一個共用資料檔案 `data.js`，所有模組從此匯入。

### 5.2 ⚠️ 藏干資料格式不一致

- `bazi.js` 使用簡單字串陣列：`'子': ['癸']`
- `lunming2.js` 使用物件陣列：`'子': [{stem:'癸', type:'main'}]`
- `geju.js` 使用結構化物件：`'子': [{stem:'癸', type:'本氣', desc:'...', power:100}]`

**建議**：統一使用一種格式，或建立轉換函數。

## 6. 安全性與錯誤處理

### 6.1 ⚠️ 輸入驗證不足（中等）

- `calculateBazi` 沒有驗證 year/month/day 範圍
- `parseInt` 沒有 radix 參數（`parseInt(el.birthYear.value)` 應為 `parseInt(el.birthYear.value, 10)`）
- 沒有防禦 XSS 的內容清洗（雖然目前內容都是自產的）

**建議**：
- 使用 `Number()` 或 `parseInt(x, 10)` 統一處理數值輸入
- 加入輸入範圍檢查
- 對使用者輸入的內容使用 `textContent` 而非 `innerHTML`

### 6.2 ⚠️ 錯誤處理不完整（中等）

部分函數有 try-catch，但有些直接 throw：

```javascript
// app.js 第 281 行 — 有 try-catch
// geju.js 無 try-catch
// lunming2.js 的 calculateDayMasterStrength 無邊界檢查
```

**建議**：為所有公開 API 加上 try-catch，並提供使用者友善的錯誤訊息。

### 6.3 ❌ 無單元測試（嚴重）

整個專案完全沒有測試檔案。核心的八字計算、五行強度分析、格局判定等邏輯都未經自動化測試驗證。

**建議**：
- 為 `calculateBazi`、`getDayPillar`、`getFavoriteElement`、`calculateGreatLuck` 等核心函數加入單元測試
- 使用 Vitest 或 Jest
- 至少測試邊界案例（立春前後、閏年、跨世紀等）
